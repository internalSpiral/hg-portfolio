Building Accessible Web Apps

H. Gwyn 6 min read

Accessibility in web development is often treated as an afterthought—but it shouldn't be. Making your web apps usable for everyone, including people with disabilities, is not just a good practice—it's essential. In this post, I’ll share what I’ve learned while making my own projects more accessible.

Why Accessibility Matters

According to the WHO, over 1 billion people live with some form of disability. Ignoring accessibility means excluding a huge portion of potential users. Plus, accessible sites are often more usable and SEO-friendly for everyone.

  • Screen reader users need semantic HTML and ARIA labels
  • Keyboard-only users depend on focus states and logical tab order
  • Color-blind users need proper contrast and non-color cues

Start with Semantic HTML

The foundation of accessibility is semantic HTML. Use proper tags for their intended purposes—<nav>, <main>, <article>, and so on. This makes your content more understandable to assistive technologies.

<button aria-label="Close modal">
            ×
        </button>

Notice the aria-label? That’s crucial when the visual element alone isn’t descriptive enough.

Keyboard Navigation

Try navigating your site using only the keyboard. Can you access all interactive elements with the Tab key? Can you activate them with Enter or Space? If not, it’s time to fix that.

1. Don’t Remove Focus Outlines

Some designers remove focus styles for aesthetics. Don’t. Instead, customize them to fit your theme without compromising usability.

2. Logical Focus Order

The tab order should follow the visual layout. This helps users navigate efficiently without confusion.

Color & Contrast

Use tools like the WebAIM Contrast Checker to ensure your color schemes meet WCAG guidelines.

  • Body text should have a contrast ratio of at least 4.5:1
  • Large text (18pt and up) can be 3:1
  • Don’t rely on color alone to convey meaning

Accessible Forms

Forms are notorious for accessibility issues. Make sure each <input> has a corresponding <label>, and use error messages that are clear and programmatically associated.

<label for="email">Email Address</label>
        <input type="email" id="email" name="email" required>

Testing and Tools

You don’t need to be an expert to get started. Tools like Axe, Lighthouse, and NVDA help catch issues early.

Browser Extensions

Use accessibility checkers right inside your dev tools. They’re not perfect, but they catch common mistakes fast.

Manual Testing

Real users with disabilities offer the best insights. Whenever possible, include them in your testing process.

Accessibility isn't about perfection—it's about progress. Every step you take makes the web more inclusive.
